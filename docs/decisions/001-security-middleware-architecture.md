# ADR 001: Security Middleware Architecture

**Status:** Accepted

**Date:** 2026-03-22

---

## Context

Registry was approaching production deployment without any security middleware
in place. Several gaps existed:

- CSRF tokens were generated by Mojolicious (via the built-in `csrf_token`
  helper) but never validated on incoming state-changing requests.
- Authentication checks were scattered across individual controller methods,
  making it easy to miss coverage and hard to audit.
- No HTTP security headers were set, leaving browsers without guidance on
  acceptable content sources, framing, and transport.
- No rate limiting existed, leaving authentication and credential-related
  endpoints open to brute-force and credential-stuffing attacks.

The question was how to introduce all four concerns in a way that is cohesive,
easy to reason about, and does not over-engineer a pre-alpha codebase.

---

## Decision

All four security concerns are implemented directly in the `startup()` method
of `lib/Registry.pm` as Mojolicious hooks and helpers, with rate limiting
extracted to a dedicated module (`lib/Registry/Middleware/RateLimit.pm`) solely
because it carries non-trivial stateful logic.

### CSRF Validation

A `before_dispatch` hook validates the CSRF token for every `POST`, `PUT`, and
`DELETE` request. Webhook routes (`/webhooks/`) are excluded because they use
their own HMAC-based authentication. The hook accepts the token from either the
`csrf_token` form field or the `X-CSRF-Token` request header, accommodating
both traditional HTML form submissions and HTMX/JavaScript fetch calls.

A companion `after_render` hook automatically injects a `<input type="hidden"
name="csrf_token">` into every `<form>` tag in HTML responses. This covers all
existing templates without requiring individual template edits.

### Security Headers

An `after_dispatch` hook appends the following headers to every response:

- `Content-Security-Policy` -- restricts resource origins; see the CSP
  discussion below.
- `X-Frame-Options: DENY` -- prevents the application from being embedded in
  frames.
- `X-Content-Type-Options: nosniff` -- prevents MIME-type sniffing.
- `X-XSS-Protection: 0` -- disables the legacy browser XSS filter (current
  guidance recommends relying on CSP instead).
- `Strict-Transport-Security` -- set only when the request arrives over HTTPS,
  either directly (`$c->req->is_secure`) or via a trusted reverse proxy
  (`X-Forwarded-Proto: https`).

The Content-Security-Policy value is built once at startup and shared across
all requests:

```
default-src 'self';
script-src 'self' 'unsafe-inline' js.stripe.com unpkg.com cdn.jsdelivr.net;
style-src 'self' 'unsafe-inline';
connect-src 'self' api.stripe.com;
frame-src js.stripe.com;
img-src 'self' data:;
font-src 'self'
```

`unsafe-inline` is required for scripts because HTMX relies on inline event
handlers (`hx-on:*` attributes and inline JavaScript). See "Alternatives
Considered" for the nonce-based approach that would eliminate this requirement.

### Rate Limiting

`Registry::Middleware::RateLimit` implements a fixed-window counter per client
key (IP address by default; uses the first IP from `X-Forwarded-For` when
present). The `before_dispatch` hook delegates to an instance of this class
that is created once at startup and shared across all requests to the same
Mojolicious worker process.

Two tiers of limits apply within a 60-second window:

- **Credential-sensitive paths** (`login`, `password`): 10 requests per window.
- **All other paths**: 100 requests per window.

Webhook (`/webhooks/`) and static asset (`/static/`, `/public/`) paths are
excluded entirely from rate limiting.

When a limit is exceeded, the middleware responds with HTTP 429 and a
`Retry-After` header indicating the seconds remaining in the current window.

### Authentication and Authorization

Authentication is centralized into two application-level helpers:

- `require_auth` -- verifies that `current_user` is present in the stash.
  Returns a 401 JSON response for API/XHR clients and redirects browsers to
  `/user-creation` otherwise.
- `require_role(@allowed_roles)` -- calls `require_auth` first, then checks
  `current_user->{user_type}` against the list of allowed roles. Returns 403
  for requests with insufficient privilege.

Role-protected sections of the route tree are guarded with Mojolicious
`under()` bridges that call these helpers:

```perl
my $admin = $r->under('/admin')->to(
    cb => sub ($c) { $c->require_role( 'admin', 'staff' ) }
);
```

Controllers no longer contain per-method authentication checks; the comment at
the top of `AdminDashboard.pm` documents that route-level guards cover the
entire controller.

## Consequences

**Positive:**

- All security enforcement for the four concerns is visible in a single file
  (`lib/Registry.pm`), making it straightforward to audit.
- Controllers are simpler and more focused; no per-method auth duplication.
- CSRF protection covers all HTML forms automatically without template changes.
- Browsers receive a consistent set of security headers on every response.

**Negative / Limitations:**

- Rate-limit counters are stored in process memory. Counters reset on server
  restart, and in a multi-process (prefork) deployment each worker process
  maintains independent counters. This is acceptable for the current single-
  process MVP but must be replaced with a shared backing store (Redis) before
  horizontal scaling or prefork deployment.
- `unsafe-inline` in the CSP weakens script-src protection against XSS. Fixing
  this requires nonce-based CSP, which in turn requires template changes across
  all files that include inline scripts (see "Alternatives Considered").
- The `startup()` method in `Registry.pm` is growing. If the number of hooks
  or helpers increases significantly, extracting them into Mojolicious plugin
  modules becomes the right next step.

**Test infrastructure:**

`t/lib/Test/Registry/Mojo.pm` provides a `Test::Mojo` subclass
(`Test::Registry::Mojo`) that automatically injects valid CSRF tokens into
`post_ok`, `put_ok`, and `delete_ok` calls. This eliminates boilerplate token-
extraction code from controller and integration tests and ensures that existing
tests are not broken by CSRF validation.

---

## Alternatives Considered

### Separate Mojolicious::Plugin modules per concern

Each of the four concerns (CSRF, security headers, rate limiting, auth) could
be packaged as its own `Mojolicious::Plugin` subclass. This would improve
modularity and allow the plugins to be reused across projects.

Rejected for the current phase because the application has exactly four hooks
and the overhead of four plugin modules, each with its own file, register
method, and test surface, is not justified. The single-file approach keeps all
security logic in one place and easy to read. If Registry evolves into a
framework or needs to share these plugins with sibling projects, extraction at
that point is straightforward.

### Redis-backed rate limiting

Using Redis as the counter store would make rate limits consistent across
multiple worker processes and across restarts, and would support rate-limit
expiry natively without a scan loop.

Deferred because Registry currently runs as a single Mojolicious process and
the MVP load profile does not require cross-worker consistency. Adding Redis as
a required infrastructure dependency before it is needed would increase
operational complexity without a corresponding benefit. The `RateLimit` module
is designed so that swapping the in-memory hash for a Redis client requires
changes only inside `Registry::Middleware::RateLimit`.

### Nonce-based Content Security Policy

A nonce-based CSP generates a unique random value per response, injects it as
an attribute on every `<script>` tag, and includes it in the CSP header. This
eliminates the need for `unsafe-inline` and is the recommended long-term
approach.

Deferred because it requires modifying every template that renders a `<script>`
tag, as well as the layout templates that include inline HTMX initialization
scripts. The scope of that change is non-trivial and the risk of missing a
script tag is high. Given that this is a pre-alpha application with no users,
the pragmatic approach is to accept `unsafe-inline` now and track nonce
migration as a future task.
