## Executive Summary

The branch fixes 14 UI/UX issues in the TinyArtEmpire tenant signup workflow. The core refactoring (pricing plan data flow, template stash reads, layout cleanup) is well-structured. Three issues require immediate attention: the Terms of Service checkbox is outside the form and therefore never submitted server-side, pricing validation errors are silently ignored due to a key name mismatch, and the pricing template has unguarded access to potentially-undef metadata. Several branding references to "registry.com" were missed in the review and payment templates.

## Critical Issues

### [C1] Terms checkbox bypass -- hidden field always sends terms_accepted=1
- **File:** `templates/tenant-signup/review.html.ep:233,256-257`
- **Bug:** The terms checkbox is outside the `<form>` tag. A hidden field inside the form unconditionally sends `terms_accepted=1`, making the server-side check at `Workflows.pm:242` a no-op.
- **Impact:** Users (or automated requests) can complete signup without agreeing to Terms of Service.
- **Suggested fix:** Move the `<form>` tag to wrap the checkbox section. Remove the hidden field. Let the checkbox's own value be submitted.
- **Confidence:** High
- **Found by:** Logic & Correctness, Security

### [C2] PricingPlanSelection validation errors silently ignored -- workflow advances past invalid selection
- **File:** `lib/Registry/DAO/WorkflowSteps/PricingPlanSelection.pm:39-43,49-54`
- **Bug:** Error returns use `errors` key, but the controller at `Workflows.pm:268` checks for `_validation_errors`. Errors are never caught. `WorkflowRun::process` merges the error hash into run data AND advances `latest_step_id`, so the workflow proceeds past pricing even on validation failure.
- **Impact:** User can reach payment step with no plan selected. Control-flow metadata (`next_step`, `errors`, `data`) pollutes the persistent run data.
- **Suggested fix:** Change error returns to use `_validation_errors` key to match the controller contract established by the base `WorkflowStep::process`.
- **Confidence:** High
- **Found by:** Contract & Integration, Error Handling

### [C3] Unguarded string comparison on potentially undef `suitable_for` metadata
- **File:** `templates/tenant-signup/pricing.html.ep:39,86`
- **Bug:** `$plan->{metadata}->{suitable_for} eq 'medium_programs'` warns on undef when a plan's metadata lacks this key. Executes for every plan card on every page load.
- **Impact:** "Use of uninitialized value" warnings in logs; could surface to users in strict mode.
- **Suggested fix:** Guard with `//`: `($plan->{metadata}->{suitable_for} // '') eq 'medium_programs'`
- **Confidence:** High
- **Found by:** Error Handling

## Important Issues

### [I1] payment.html.ep reads stash('data') but controller spreads keys directly into stash
- **File:** `templates/tenant-signup/payment.html.ep:4-8`
- **Bug:** Template does `stash('data')->{billing_summary}` but `TenantPayment::prepare_template_data` returns `{ billing_summary => ..., ... }` which gets spread into the stash as individual keys. `stash('data')` is always `{}`.
- **Impact:** Payment page renders with empty billing info, no plan details, no org name. Pre-existing bug, same pattern as the pricing/review stash fix in this branch.
- **Suggested fix:** Update template to read `stash('billing_summary')` etc., or wrap return in `{ data => ... }`.
- **Confidence:** High
- **Found by:** Contract & Integration

### [I2] get_organization_preview can crash on undef if latest_run returns nothing
- **File:** `lib/Registry/DAO/WorkflowSteps/PricingPlanSelection.pm:148-158`
- **Bug:** `$workflow->latest_run($db)` can return undef. Line 151 then calls `$run->data` on undef, causing a fatal error. Also wastes two queries re-fetching a `$run` that was already available as a parameter to `prepare_template_data`.
- **Impact:** 500 error on pricing step if called with an unexpected workflow state.
- **Suggested fix:** Guard against undef, or pass `$run` through from `prepare_template_data`.
- **Confidence:** High
- **Found by:** Error Handling, Contract & Integration

### [I3] Hardcoded "30-Day Free Trial" heading while trial_days is dynamic
- **File:** `templates/tenant-signup/review.html.ep:154,157`
- **Bug:** Heading says "30-Day Free Trial" but date calculation uses `$plan_config->{trial_days} || 30`. A plan with 14-day trial shows "30-Day" heading with a 14-day date. `|| 30` also treats `trial_days => 0` as falsy.
- **Impact:** Confusing/incorrect information during signup review.
- **Suggested fix:** Make heading dynamic: `<%= $trial_days %>-Day Free Trial`. Use `//` instead of `||`.
- **Confidence:** High
- **Found by:** Error Handling
