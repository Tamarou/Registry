## Executive Summary

Strong progress on workflow coverage, engine fixes, and hardening tests. Two blockers must be fixed: the `error`/`errors` key mismatch in Submit steps silently swallows failures, and blessed DAO objects in Submit step returns crash JSON serialization. The `latest_run` concurrency issue is architecturally significant but acceptable for pre-alpha. The N+1 query pattern in ProgramListing needs addressing before production traffic.

## Critical Issues

### [C1] error/errors key mismatch -- Submit step failures silently swallowed
- **File:** `lib/Registry/DAO/WorkflowSteps/SubmitDropRequest.pm:23,40` and `SubmitTransferRequest.pm:23,38`
- **Bug:** Returns `{error => "..."}` (singular) but engine only checks `errors` (plural). Error is persisted as domain data and step advances as if successful.
- **Impact:** Failed drop/transfer submissions appear to succeed. User sees completion page but enrollment is not actually dropped.
- **Fix:** Change to `return { errors => [$message] }` to match convention
- **Found by:** Logic, Error Handling, Security, Code Quality, Contract (5 specialists)

### [C2] Blessed objects in Submit step returns crash JSON serialization
- **File:** `SubmitDropRequest.pm:33` and `SubmitTransferRequest.pm:31`
- **Bug:** `drop_request` and `transfer_request` DAO objects stored in return hash. Not in TRANSIENT_KEYS, so `to_json()` tries to serialize blessed objects and dies.
- **Impact:** Non-immediate drop requests crash the workflow at the serialization step.
- **Fix:** Store `drop_request_id => $result->{drop_request}->id` instead of the object
- **Found by:** Logic, Contract (2 specialists)

## Important Issues

### [I1] latest_run returns wrong run under concurrency
- **File:** 9 step classes, all do `$workflow->latest_run($db)`
- **Bug:** Returns most recently created run globally, not the current user's. Under concurrent usage, User A's step reads User B's data.
- **Impact:** Data leakage between users. Acceptable for pre-alpha with no concurrent users, but must be fixed before production.
- **Fix:** Pass the run object from `WorkflowRun::process` to `step->process` as a third argument
- **Found by:** Concurrency, Logic, Error Handling, Contract, Code Reuse, Code Quality (6 specialists)

### [I2] SQL injection vector in count_for_session
- **File:** `lib/Registry/DAO/Enrollment.pm:163`
- **Bug:** `$statuses` interpolated into SQL with manual quoting. No current caller passes user input, but the interface allows it.
- **Fix:** Use parameterized `ANY(?)` or placeholder expansion
- **Found by:** Logic, Efficiency (2 specialists)

### [I3] Stale template_data references in CollectDropReason/CollectTransferReason
- **File:** `CollectDropReason.pm:31-32`, `CollectTransferReason.pm:27-28`
- **Bug:** References `$run_data->{enrollment}` and `$run_data->{family_member}` but upstream now stores `enrollment_id` and `child_name`. Template data will be undef.
- **Fix:** Update to use new key names or load from IDs
- **Found by:** Logic, Error Handling (2 specialists)

### [I4] N+1 queries in ProgramListing (~120 queries per storefront page)
- **File:** `lib/Registry/DAO/WorkflowSteps/ProgramListing.pm:34-87`
- **Bug:** Up to 6 queries per session in a loop. Public-facing storefront page.
- **Fix:** Consolidate into 1-2 queries with JOINs
- **Found by:** Concurrency, Efficiency (2 specialists)

## Suggestions
