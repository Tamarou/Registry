## Executive Summary

Strong test infrastructure work with 12 production bug fixes, but the review found 4 critical issues in the production code changes: a client-supplied user_id allowing account takeover, accept_offer using wrong status value, SQL injection in process_waitlist, and a broken template call. The demo mode payment bypass and data shape inconsistency between payment paths are also important to address before merging.

## Critical Issues

### [C1] Client-supplied user_id in continue_logged_in allows account takeover
- **File:** `lib/Registry/DAO/WorkflowSteps/AccountCheck.pm:71`
- **Bug:** `$form_data->{user_id}` is read from POST body, allowing any user to impersonate another by submitting an arbitrary user_id
- **Impact:** Full identity impersonation within registration workflow -- enrollment fraud, potential payment fraud
- **Suggested fix:** Only use `$run->data->{user_id}`, never trust form data for identity
- **Confidence:** High
- **Found by:** Security

### [C2] accept_offer sets status to 'declined' instead of 'accepted'
- **File:** `lib/Registry/DAO/Waitlist.pm:211-213`
- **Bug:** Both accept and decline write `status => 'declined'`, making them indistinguishable. The ADJUST validator also lacks an 'accepted' status.
- **Impact:** Data integrity -- cannot audit or query which offers were accepted vs declined; admin dashboard conflates them
- **Suggested fix:** Add 'accepted' to ADJUST regex, use `status => 'accepted'` in accept_offer
- **Confidence:** High
- **Found by:** Logic, Concurrency, Error Handling (3 specialists)

### [C3] SQL injection in process_waitlist via string interpolation
- **File:** `lib/Registry/DAO/Waitlist.pm:89`
- **Bug:** `$hours_to_respond` interpolated directly into SQL: `"SELECT NOW() + INTERVAL '$hours_to_respond hours'"`
- **Impact:** Latent SQL injection -- currently safe (internal callers, default 48), but any future untrusted input to this parameter is exploitable
- **Suggested fix:** Use parameterized query: `"SELECT NOW() + ? * INTERVAL '1 hour'", $hours_to_respond`
- **Confidence:** High
- **Found by:** Concurrency, Error Handling (2 specialists)

### [C4] offer_is_active called without required $db argument in parent_status template
- **File:** `templates/waitlist/parent_status.html.ep:60,78`
- **Bug:** Template calls `$entry->offer_is_active` but method requires `($db)` parameter -- will crash at runtime
- **Impact:** Parent waitlist status page is broken
- **Suggested fix:** Pass $db to method, or pre-compute the flag in the controller and pass via stash
- **Confidence:** High
- **Found by:** Logic

## Important Issues

### [I1] Payment paths use different data shapes -- real Stripe path reads unpopulated 'children' key
- **File:** `lib/Registry/DAO/WorkflowSteps/Payment.pm:183 vs 124`
- **Bug:** Demo path reads `enrollment_items`, real Stripe path reads `children`+`session_selections`. The `children` key is never populated by MultiChildSessionSelection, so real Stripe payments would create zero enrollments.
- **Impact:** Money collected but no registrations recorded when Stripe is enabled
- **Suggested fix:** Unify both paths to use `enrollment_items`
- **Confidence:** High
- **Found by:** Logic, Error Handling, Contract, Code Quality (4 specialists)

### [I2] time() vs PostgreSQL NOW() mismatch in waitlist management query
- **File:** `lib/Registry/DAO/Waitlist.pm:417`
- **Bug:** Perl `time()` epoch integer compared against PostgreSQL `timestamptz` column
- **Impact:** Urgent waitlist filter returns wrong results
- **Suggested fix:** Use SQL-side time computation: `NOW() + INTERVAL '24 hours'`
- **Confidence:** High
- **Found by:** Logic, Error Handling (2 specialists)

### [I3] user_type privilege escalation via form data
- **File:** `lib/Registry/DAO/WorkflowSteps/AccountCheck.pm:34`
- **Bug:** `user_type => $form_data->{user_type} || 'parent'` -- attacker can POST `user_type=admin`
- **Impact:** Privilege escalation if admin/staff roles grant elevated access
- **Suggested fix:** Hard-code `user_type => 'parent'`
- **Confidence:** High
- **Found by:** Security

### [I4] Internal exception details leaked to users in Waitlist controller
- **File:** `lib/Registry/Controller/Waitlist.pm:94,97,140,142`
- **Bug:** Raw `$e` included in flash messages and JSON error responses
- **Impact:** Stack traces, SQL errors, file paths exposed to clients
- **Suggested fix:** Log full error server-side, return generic message to user
- **Confidence:** High
- **Found by:** Security

## Suggestions
